注册功能

注册流程

BFF层服务webchat-sso

为什么要在这个服务里面校验验证码，而不是再下游统一校验；这主要还是因为session再每一个服务不会共享；验证码的生成是保存再webchat-sso服务里面的，如果去取下游的服务去校验验证码，下游的session里面是肯定没有存储的，所以必然会校验失败。

正确的做法是验证码校验必须在 webchat-sso 服务内完成，而不下沉到业务服务中。
原因在于验证码属于认证前置状态，其生命周期仅存在于身份建立之前，应由统一的认证入口负责校验。
在微服务架构下，各服务的 Session 天然隔离，验证码生成状态仅存在于 webchat-sso 的会话上下文中，下游服务既无法也不应直接访问该状态。
若将验证码校验下沉到业务服务，将导致跨服务状态依赖和认证逻辑扩散，破坏服务边界并引入安全风险。因此，验证码校验统一在webchat-sso内完成，校验通过后再进入后续注册/登录流程。

二、单点登录

单点流程

登录实现方法

有了Token为什么还需要授权码？Token和授权码是什么？

简单来说Token是登录凭证，授权码是用来换取登录的临时凭证；

Token的本质上来说是已经登录用户的会话凭证，特点是和userId强绑定、生命周期较长、每次请求都会携带、可以根据Token直接点各位用户；通常用来做API鉴权、用户态维持、SSO统一登录

授权码的本质，我已经通过认证的一次性证明；特点是只能用一次（按照项目的需求来，一般就能用一次）、快速过期、不能直接访问业务接口、只能用来换取Token；用途，SSO跨系统登录，避免token暴露给前端/URL。

如果只用Token会有什么问题？

问题1、token容易泄露，会暴露再URL参数，Header头、Referer中

问题2、没有办法支持SSO跳转，A系统跳转到B系统Token怎么进行安全传递

问题3、Token被重放的风险比较高，一旦Token被劫持就可以长期使用

维度 token 授权码
是否一次性 ❌ ✅
是否短期 ❌ ✅
是否可直接访问接口 ✅ ❌
是否可暴露在 URL ❌ ✅
是否可重放 高风险 极低

总结：Token 是用户的登录态凭证，用于后续所有请求鉴权；登录授权码是一次性的中转凭证，只用于安全地换取 token，本身不具备访问能力。

一个典型SSO/OAuth时序

  • ① 用户在 SSO 登录
  • ② SSO 校验账号密码
  • ③ SSO 生成 token（服务器内部）
  • ④ SSO 返回 loginOauthCode 给前端
  • ⑤ 前端跳转 / 回调到业务系统，带 code
  • ⑥ 业务系统用 code 去 SSO 换 token
  • ⑦ 业务系统拿 token → 建立登录态

四、总结

验证码设计

    验证码属于认证前置状态，仅与当前用户会话相关
       不需要跨用户、跨服务共享，使用 Session 存储是合理的
       校验必须在统一认证入口（BFF / SSO）完成，不能下沉到业务服务

Session 与分布式

分布式环境下 Session 天然隔离，不可依赖下游服务的 Session 状态
sessionId + key 共同构成隔离空间，不存在多用户覆盖问题

事务与缓存一致性

事务未提交前的数据不具备对外可见性
事务内直接刷新缓存可能导致：
      RC 隔离级别下缓存写入空值
      事务回滚但缓存已写，产生脏数据
缓存写入应延迟到事务成功提交之后

Token 与授权码

Token 是登录态凭证，用于请求鉴权和用户身份识别
授权码是一次性、短生命周期的中转凭证
授权码用于安全地换取 Token，避免 Token 暴露和重放风险

SSO 认证流程

登录态优先级：业务系统本地登录态 → SSO Token → 授权码
授权成功后在业务系统建立本地 Session，避免每次请求依赖 SSO
认证失败统一重定向到 SSO 登录入口

架构边界原则

认证状态集中管理，避免跨服务状态依赖
状态只在必要范围内存在，生命周期尽量短
核心目标是减少状态扩散，降低系统耦合和安全风险
